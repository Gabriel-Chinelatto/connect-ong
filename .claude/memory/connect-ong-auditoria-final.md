## ✅ REVISÃO FINAL DE SEGURANÇA (v1.8) — 2026-07-10 (fecha os achados restantes + extras)
Segunda rodada do mesmo dia, marcada como **v1.8 "Revisão final de segurança"** no changelog (`data/versoes.dart` dos 2 apps, `atual: true`; v1.7 deixou de ser atual). pubspec fica 1.0.0+1 (metadado de build; versionamento de produto = changelog). Backend **135 testes verdes**, apps analyze limpo. Commits: backend `031fdcf` (master), mobile `1977efd`, desktop `8a0fde1` (main), todos com push.
- **Achado 3 (XFF) RESOLVIDO:** novo `security/ClientIpResolver` (ponto único). Rate limiting (RateLimitService) e auditoria (AuditService) NÃO confiam mais no X-Forwarded-For por padrão (era forjável com a API exposta direto → burlava anti-brute-force/limites por IP e falseava o IP do audit). Agora usam getRemoteAddr(); confiar no XFF é opt-in `app.proxy.trust-forwarded-for=true` (só atrás de proxy reverso confiável). RateLimitTest já usava setRemoteAddr, então passou sem mudança.
- **Achado 7 (POST /ongs legado) RESOLVIDO por REMOÇÃO:** endpoint + `ONGService.criar` deletados (criavam ONG fantasma sem dono; nenhum frontend usava — desktop usa GET /ongs e o cadastro real é /ongs/registro; nenhum teste dependia).
- **Nota /contribuir RESOLVIDA:** `CampanhaService.contribuir` agora deriva o nome do autor do TOKEN (`nomeDoAutenticado()` por security.usuarioId()), ignorando o `doadorNome` do corpo (era spoofável na notificação) + rate limit generoso por IP (`app.ratelimit.max-contribuicoes`, default 30).
- **Extra do agente de revisão (achado 1) RESOLVIDO:** `DoacaoFinanceiraService.doar` (PIX SIMULADO, `/doacoes-financeiras`) era um 2º caminho para inflar/auto-encerrar campanha de qualquer ONG sem rate limit → adicionado rate limit por IP (`app.ratelimit.max-doacoes`, default 30). - **Extra do agente (achado 3) RESOLVIDO:** endpoint público `/assistente` ganhou `@Size(max=20)` na lista `historico` + `@Valid` em cascata + `@Size` no texto/papel de cada item (evita corpo gigante anônimo pressionando memória/tokens da IA).
- **Robustez de teste:** `ContratosFeiraTest.pendencias_...` era FLAKY (diasRestantes 8↔9: `now().minusDays(2)` na fronteira de 48h + truncação do H2 fazia `DAYS.between` oscilar). Tornado determinístico com margem `.minusHours(1)` (semântica e valor esperado 8 inalterados). LIÇÃO: evitar timestamps exatamente na fronteira de dia em asserts de contagem de dias.

## 🟡 DECISÕES CONSCIENTES ADICIONAIS (da revisão final — documentar, não "corrigir" agora)
- **Review bombing (agente, MÉDIA):** avaliar ONG/doador não exige um match CONCLUÍDO entre o par (o autor NÃO é spoofável — vem do token — mas qualquer doador avalia qualquer ONG e vice-versa; nota entra em notaMedia pública/ranking). Não corrigido nesta rodada: exigir Interesse concluído é mudança de regra de negócio com risco de quebrar a demo/testes da feira. Para pós-feira. (Uma conta por review + contas fáceis = mitigação fraca, ciente.)
- **Anexos/base64 sem validar bytes reais (agente, BAIXA):** `anexoTipo` é regex-limitado a "imagem" mas o base64 em si não é validado (poderia ser text/html ou SVG). Risco real BAIXO: os clientes são Flutter (Image.memory decodifica bytes de imagem, não executa script); XSS só existiria num renderizador HTML. Aceito p/ o escopo TCC; hardening futuro = validar magic-bytes de imagem (também em PrestacaoService/foto de perfil).

## 🔴 FALHAS REAIS — RESTANTES (não aplicadas neste pacote)

Intencional p/ a feira, mas o DEFAULT é perigoso. FIX: inverter default p/ `false` e ligar só na máquina da feira (o `app.security.enforce` já é true por padrão — aplicar o mesmo critério).
2. ✅ RESOLVIDO (ver pacote acima). **[MÉDIA] Privacidade de contato contornável** — `GET /ongs` (ONGService.listar/toDTO :236-264,:420-433) e `GET /ongs/{id}` (obterPorId/toDTORico :306-312) devolvem email/telefone/cnpj de qualquer ONG p/ qualquer autenticado, IGNORANDO os toggles mostrarEmail/mostrarTelefone (só o perfil-publico respeita, via `aplicarPrivacidade` :144-165). FIX: aplicar `aplicarPrivacidade`/omitir contato também nesses 2 caminhos. **(quick win)**
3. ✅ RESOLVIDO na v1.8 (ver acima). **[MÉDIA] Rate limit + IP de audit confiam em X-Forwarded-For sem proxy** — RateLimitService.java:151-153 (e AuditService). API exposta direto → cliente forja XFF e burla anti-brute-force do login (chave email+IP, UsuarioService.java:184), limite de cadastro/esqueci-senha/cota da IA, e falseia IP no audit-log. FIX: usar getRemoteAddr() sem proxy; com proxy, aceitar XFF só do IP do proxy.
4. ✅ RESOLVIDO (ver pacote acima). **[MÉDIA] Mobile trava com token expirado** — sem tratamento GLOBAL de 401. `main.dart:147` (SplashDecider) decide rota só pelo UsuarioLogado persistido; token vencido → entra no MainShell e tudo dá 401 sem deslogar (só há 401 pontual em configuracoes_screen.dart:486 e perfil_service.dart:83). FIX: interceptar 401 no ApiService._executar → SessionService.logout() + volta ao login. **(quick win, alto valor p/ estabilidade na feira)**
5. ✅ RESOLVIDO (ver pacote acima). **[MÉDIA] `fromJson` frágil nos 2 apps (int×bool / null)** — models fazem `id: json['id']` sem cast e `urgente: json['urgente'] ?? false` (mobile models/usuario_logado.dart:29-35, necessidade; desktop models/necessidade.dart:25). MESMA classe do bug do `doisFatores` (backend manda 0/1, app espera bool → crash). FIX: casts tolerantes (`(json['id'] as num?)?.toInt() ?? 0`; `json['urgente']==true || json['urgente']==1`). **(quick win)**
6. ✅ RESOLVIDO (ver pacote acima). **[BAIXA] `/auth/refresh` não checa soft-delete** — AuthController.java:99-108 só faz findById; conta excluída segue emitindo access token por 7 dias via refresh (login/2FA/reset já barram soft-delete).
