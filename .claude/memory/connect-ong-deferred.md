## Segurança / conformidade
- **[x] JWT enforcement** (Bloco 15 → FEITO 2026-06-26): ATIVADO. Spring Security real no backend (`SecurityFilterChain` + `JwtAuthFilter` validando o Bearer), whitelist pública (`/auth`, `/publico`, swagger, `POST /usuarios`, `/usuarios/login`, `/ongs/registro`), CORS central (não mais `*`), JWT secret via `app.jwt.secret`/env, senha `WRITE_ONLY`, DTOs com `@Valid` no cadastro/login (fecha mass assignment), login 401 genérico. Os 3 apps enviam `Authorization: Bearer` (mobile persiste token em SharedPreferences; desktop em memória pois não persiste sessão). Toggle `app.security.enforce` (default true) = válvula de escape p/ feira. Access token = 12h.
- **[x] Autorização por DONO (ownership)** (FEITO 2026-06-26): principal `UsuarioAutenticado` (id/tipo/ongId do token) + `SecurityUtils.exigirUsuario/exigirOng/exigirUsuarioOuOng` + `AcessoNegadoException`→403. Aplicado em perfil/senha/preferências, notificações, favoritos, doações financeiras, interesses (listar/demonstrar e aceitar/recusar só pela ONG dona), avaliar, e chat (só participantes do match). Apps NÃO precisaram mudar (já mandam o próprio id = id do token; usuário-ONG tem ongId, inclusive nos dados de demo). Provado por `SecurityEnforcementTest` (8 testes: acesso a perfil/notificação de outro → 403). **Pendente futuro (nice-to-have):** `@PreAuthorize` por papel.
- **[ ] Senha do banco no histórico do git** (sessão inicial): a senha foi externalizada para `application-local.properties` (gitignored), MAS commits antigos ainda contêm a senha em texto puro no histórico. **AÇÃO DO USUÁRIO, NÃO FEITA:** rotacionar a senha do MySQL da escola (`cl*28032008` @ 143.106.241.3) E limpar o histórico (BFG/git filter-repo). Idem definir `APP_JWT_SECRET` novo em prod.
- **[x] JWT secret sem default hardcoded** (FEITO 2026-07-02): `JwtService` agora usa `@Value("${app.jwt.secret}")` SEM default — a app FALHA no startup sem a chave (antes um default público no código permitia forjar tokens). Chave forte adicionada ao `application-local.properties` (gitignored, dev) e placeholder no `.example`. Commit backend `1497be0`.
- **[x] IDOR corrigido em massa** (FEITO 2026-07-02, commits `fac1ecb`+`fa5410f`): antes qualquer logado podia apagar/editar/**auto-verificar QUALQUER ONG**, publicar necessidades/campanhas em nome de ONGs alheias, encerrar campanhas de terceiros e ler prestações de outros; audit-logs e denúncias eram lidos por qualquer um. Agora: `SecurityUtils.exigirOng(...)` em ONG(atualizar/deletar/verificar)/Necessidade(criar)/Campanha(criar/encerrar), `exigirParticipante` em Prestação(listar/criar), `/audit-logs` e `GET|PUT /denuncias` restritos a `ROLE_ONG`, mass assignment no `POST /ongs` zerado (verificada/notaMedia), `@Size(max=3MB)` em fotoBase64, senha padronizada em min 6. +2 testes (25/25 passam). **Pendente futuro:** verificar-ONG idealmente deveria ser `ROLE_ADMIN` (hoje ainda permite auto-verificação da própria ONG, pois não há conceito de admin) — criar papel admin.
- **[x] Rate limiting no login + mitigação de enumeração** (FEITO 2026-07-03, commit backend `d54b034`): `RateLimitService` in-memory (sem dependência nova) — login 5 falhas consecutivas por email+IP → 429 por 15 min (sucesso zera); esqueci-senha e cadastros públicos máx 5/15min por IP; 5 códigos errados no redefinir-senha → 400 genérico sempre. Configurável via `app.ratelimit.*` (desligado no properties de TESTE — MockMvc compartilha IP). Enumeração de e-mail: mensagem "Email já cadastrado" MANTIDA pela UX da feira; mitigada pelo limite por IP (decisão registrada em comentário no código).
- **[x] 2ª rodada rigorosa de testes+segurança** (FEITO 2026-07-02, commit backend `2eca82b`, desktop `6dc3af8`, mobile `c333dee`): subi a API local com H2 e exercitei cada endpoint via HTTP forçando erros. **Corrigido e verificado ao vivo (17/17) + 27 testes:** (A1) `POST /usuarios` forçava `tipo` do cliente → qualquer um virava ONG com JWT ROLE_ONG; agora força DOADOR (só `/ongs/registro` cria ONG). (A4) `GET /interesses` sem filtro caía em `findAll()` e vazava TODOS os matches; agora exige doadorId/ongId com checagem de dono. (M1) remetente do chat vinha do corpo (spoofing) → agora derivado do token. (M2/A5) denunciante vinha do corpo → agora do token, e `denuncianteId` removido da resposta (denúncia anônima). (M4) doador do match publicava prestação como se fosse a ONG → agora só a ONG dona. (M5) listagem de doações da ONG vazava `codigoPix` do doador → omitido. **Bug funcional:** cadastro de ONG sem telefone/cidade falhava vazando ConstraintViolation interno (entidade `Ong` exigia `@NotBlank` mas form/DTO os tratam como opcionais) → entidade relaxada + coalesce null→"" + handler de ConstraintViolation não vaza mais internals; +handler `DataIntegrityViolationException`→400; senha ONG min 6.
- **[x] ROLE_ADMIN dedicado** (FEITO 2026-07-02, commit backend `13cfafd`): criado o papel **ADMIN não auto-provisionável**. `AdminBootstrap` (ApplicationRunner) cria a conta ADMIN no startup a partir de `app.admin.email`/`app.admin.password` (env `APP_ADMIN_*` em prod; `application-local.properties` gitignored no dev) — idempotente e, se vazio, NÃO cria nada (sem backdoor). Não há rota para se cadastrar como admin. No `SecurityConfig`, `/audit-logs`, `GET|PUT /denuncias` e `PUT /ongs/{id}/verificar` passaram de `ROLE_ONG` → `ROLE_ADMIN` (fecha: audit-log com IP/emails só p/ admin; só admin modera denúncia; selo de verificação vira ato admin, fim da auto-verificação). `ONGService.verificar` sem checagem de dono (admin não tem ongId). `/demo/seed` continua autenticado (o desktop usa no "Modo Feira") mas atrás do interruptor `app.demo.enabled` (default true; `APP_DEMO_ENABLED=false` desliga em prod) — com ADMIN, ser ONG demo não dá mais poder especial. Verificado ao vivo (H2) + 8 testes novos (**35 no total**): doador/ONG→403, ADMIN→200 em audit/denúncia/verificar; login do admin provisionado → tipo=ADMIN; admin verifica ONG (verificada=true); demo desligado→403. **Para usar em prod:** definir `APP_ADMIN_EMAIL`/`APP_ADMIN_PASSWORD` (e `APP_DEMO_ENABLED=false`).
- **[ ] IDOR residual em doações legadas sem dono** (A6, 2026-07-02): `DoacaoService.atualizar/deletar` deixam itens com `doadorId==null` editáveis/removíveis por qualquer logado (decisão consciente: o catálogo global do painel desktop usa itens sem dono). Baixo risco; se fechar, fazer backfill de dono antes para não quebrar o catálogo.
- **[x] NumberFormatException→500** (B2 — FEITO 2026-07-03, commit backend `381eb0e`): `POST /favoritos` e `POST /campanhas/{id}/contribuir` agora usam DTOs tipados (`FavoritoRequestDTO`, `ContribuicaoRequestDTO`) com `@Valid @NotNull @Positive` → entrada não numérica vira 400 (era 500). Mesmos nomes de campo (compatibilidade com os apps provada por teste).

- **[x] Exclusão segura de contas (soft-delete)** (FEITO 2026-07-02, commit backend `769c838`): a preocupação "excluir doador/ONG corrompe/deixa órfãos?" foi resolvida com SOFT-DELETE. Colunas `usuario.data_exclusao` e `ong.data_exclusao` (datetime; null=ativo). Nada é apagado fisicamente (sem erro de FK, sem órfãos, histórico preservado). `DELETE /ongs/{id}` marca ONG+conta; novo `DELETE /usuarios/{id}` (self, exigirUsuario) marca a conta (+ONG se for ONG). Conta excluída não loga (401 genérico); ONG some de /ongs, feed /necessidades, ranking, perfil-publico/transparencia (404); estatísticas contam só ativos. 40 testes. **Limitação:** token já emitido vale até expirar (~12h); sem revogação de token ativo. (Reduz a urgência do item de FK acima: pela app nunca há exclusão física.)
